Scope and encodings covered
RFC 4648 specifies the Base16, Base32, and Base64 data encodings, including canonical alphabets, padding rules, and permissible variations (including base64url variant). Implementations should follow the exact alphabets and padding semantics when interoperability is required.

Base64 alphabet and exact mapping
- Base64 alphabet (index 0..63):
  ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
- Padding character: '=' (used to fill the final output to a 4-character quantum when input length mod 3 != 0).
- Base64url variant alphabet (index 0..63):
  ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_
  base64url typically omits padding for URL-safe usage; when omitted, decoders must reconstruct final quantum based on string length.

Base64 encoding algorithm (24-bit quantum, 6-bit indices)
1. Take the input byte stream and process in groups of 3 bytes (24 bits).
2. Split each 24-bit group into four 6-bit values (indexes 0..63).
3. Map each 6-bit index to a character in the chosen alphabet and emit four output characters per 3 input bytes.
4. If the final quantum contains fewer than 3 bytes: pad the input with zero bits to form a 24-bit group, compute the resulting 6-bit indices, map to characters, and then replace the appropriate number of output characters with '=' padding (1 or 2 '=' characters) so the total output length is a multiple of 4.

Padding rules and no-padding variants
- Standard base64: padding is required to make the output length a multiple of 4, using '=' characters.
- base64url: padding may be omitted in many contexts; when omitted, decoders must compute the number of missing bytes by examining length mod 4 and append zero bits as needed.
- Decoding must reject illegal characters and produce an error for invalid padding placements.

Decoding algorithm and length computation
1. Map each input character to its 6-bit index via a lookup table.
2. Accumulate 24-bit groups by concatenating four 6-bit values; then split into three output bytes.
3. When padding is present: treat '=' as absent bits and compute the resulting output byte count accordingly (1 or 2 output bytes from a short quantum).
4. When no padding is present (base64url without '='), compute expected output length as floor((input_length * 6) / 8) and strip any zero-padding bits introduced in the final quantum.

Bits per character and density
- Base64: charset size 64, bits per char = log2(64) = 6 bits/char.
- For 128-bit UUID the raw base64 (no padding) length is 22 characters (128 / 6 = 21.333 => 22 chars when using 22*6=132 bits with trailing unused bits trimmed by spec behavior).

Reference details and API patterns
- encode(input: Uint8Array, padding?: boolean, urlSafe?: boolean) -> string
  - input: Uint8Array containing binary data
  - padding: boolean (default true) — whether to emit '=' padding characters
  - urlSafe: boolean (default false) — if true, use base64url alphabet and optionally omit padding
- decode(input: string, urlSafe?: boolean) -> Uint8Array
  - input: base64 or base64url string (with or without padding if urlSafe is true)
  - returns: Uint8Array of decoded bytes
- Decoders MUST validate input length and character set and throw/return an error for invalid sequences.

Supplementary implementation notes
- Use lookup tables for mapping characters <-> 6-bit indexes for performance.
- For JavaScript: use TextEncoder/TextDecoder when converting to/from UTF-8 strings and Uint8Array; use Buffer.from(str, 'base64') in Node.js when available for performance.
- For no-padding base64 (URL-safe), compute output byte length from input length using floor((len * 6)/8) to allocate output buffers.
- Security note: when comparing decoded values in sensitive contexts use constant-time comparison.
